# README 模板

在编写或更新项目 `README.md` 时使用本文件。

内容保持简练、可执行、可交付。
始终保留同一份 `README.md`，计划版、交付版、补丁版都在这份文件里更新，不再拆新文件。

## 计划阶段怎么写

- `任务目标`、`数据简述` 必写
- `路径结构和实现功能` 优先按当前项目已有目录、脚本和文件命名写；若项目尚无约定，再按默认工作流写预期目录和脚本分工
- `实现算法与细节` 只有在算法不直观时再写
- `可能存在的问题` 只有在确实有风险时再写
- `图片简述` 先写：`待执行后补充。`
- `后续版本补充` 先留空或写 `待后续更新。`

## 交付阶段怎么写

- 保留原有结构，按需补全 `路径结构和实现功能`、`实现算法与细节`、`可能存在的问题`
- `路径结构和实现功能` 优先写当前项目实际结构；若项目原本没有既有布局，再按 skill workflow 写清 `data_processed/`、`fig1/compute_*.py`、`fig1/plot_*.py`、`fig1/fig1_*.png`、`fig1/fig1_*.svg`
- 把 `图片简述` 补完整
- 把 `后续版本补充` 写入本次交付说明

## 后续修改怎么写

- 不重写整份 `README.md`
- 只在 `后续版本补充` 下追加新条目
- 若图有明显变化，同步更新 `图片简述`
- 小修图且不改科学方法时，不必强行补出 `data_processed/` 或 `compute_*.py`
- 若项目已有 `scripts/`、`figures/`、`paper_figs/` 等结构，按实际路径记录，不要为了套模板而重构目录
